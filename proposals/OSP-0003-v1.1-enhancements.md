# OSP-0003: OSOP v1.1 — Workflow Composition & Advanced Flow Control

**Status:** Draft
**Authors:** Archie (OSOP Core)
**Created:** 2026-04-02
**Target:** osop_version "1.1"
**Schema:** v2.1.0

---

## Abstract

This proposal introduces five enhancements to the OSOP protocol that address the most-requested workflow modeling gaps: sub-workflow composition, foreach iteration, explicit join modes, switch/case branching, and workflow-level timeouts. All additions are backwards-compatible — existing v1.0 files remain valid.

---

## Motivation

OSOP v1.0 handles ~70% of real-world workflow patterns. Analysis of 124 use cases across 25+ industries revealed five recurring patterns that cannot be expressed cleanly:

1. **Modular composition** — Large workflows cannot be decomposed into reusable parts
2. **List iteration** — Loop edges only support `while` conditions, not `for-each` over collections
3. **Flexible parallelism** — All parallel branches must complete (no "first wins" or "N-of-M" patterns)
4. **Multi-way routing** — Multiple conditional edges are verbose for switch/case patterns
5. **Global deadlines** — Only per-node timeouts exist; no way to set a workflow-wide deadline

---

## 1. Workflow-Level Timeout

### Specification

Add `timeout_sec` (optional, number, minimum 0) to the workflow root level.

```yaml
osop_version: "1.1"
id: "daily-report"
name: "Daily Report Generation"
timeout_sec: 3600  # 1 hour deadline for entire workflow
```

### Semantics

- If the total elapsed time exceeds `timeout_sec`, the workflow transitions to `TIMED_OUT` status.
- In-progress nodes receive a cancellation signal.
- Per-node `timeout_sec` values are still respected independently.
- If both workflow and node timeouts are set, the shorter one applies.

### Schema Change

```json
"timeout_sec": { "type": "number", "minimum": 0 }
```

Added to root `properties` in `osop.schema.json`.

## 2. foreach Iteration

### Specification

Add two optional fields to edge definitions when `mode: "loop"`:

- `for_each` (string) — CEL expression resolving to a list
- `iterator_var` (string) — Variable name for the current item

```yaml
edges:
  - from: "get_users"
    to: "send_email"
    mode: "loop"
    for_each: "outputs.get_users.user_list"
    iterator_var: "user"
```

### Semantics

- The target node executes once per item in the list.
- The current item is accessible as `${loop.<iterator_var>}` in the target node's runtime config.
- If `for_each` is set, `when` is ignored (they are mutually exclusive for loop mode).
- Empty list = zero iterations (target node is skipped).
- `max_iterations` can still be used as a safety limit.

### Schema Change

```json
"for_each": { "type": "string" },
"iterator_var": { "type": "string" }
```

Added to `$defs/edge/properties`.

## 3. Explicit Join Modes

### Specification

Add two optional fields to edge definitions:

- `join_mode` (enum: `wait_all`, `wait_any`, `wait_n`) — default: `wait_all`
- `join_count` (integer, minimum 1) — required when `join_mode: "wait_n"`

```yaml
edges:
  - from: "search_google"
    to: "merge_results"
    mode: "parallel"
    join_mode: "wait_n"
    join_count: 2
  - from: "search_bing"
    to: "merge_results"
    mode: "parallel"
  - from: "search_brave"
    to: "merge_results"
    mode: "parallel"
```

### Semantics

| Mode | Behavior |
|---|---|
| `wait_all` | Target executes after **all** parallel sources complete. Default (v1.0 behavior). |
| `wait_any` | Target executes after the **first** source completes. Other branches are cancelled. |
| `wait_n` | Target executes after `join_count` sources complete. Remaining are cancelled. |

- `join_mode` is only meaningful on edges where the target has multiple incoming parallel edges.
- It is specified on any one of the incoming edges (not all — first found wins).
- Cancelled branches transition to `SKIPPED` status in the execution log.

### Schema Change

```json
"join_mode": { "type": "string", "enum": ["wait_all", "wait_any", "wait_n"], "default": "wait_all" },
"join_count": { "type": "integer", "minimum": 1 }
```

Added to `$defs/edge/properties`.

## 4. Switch/Case Branching

### Specification

Add a new edge mode `switch` with two supporting fields:

- `cases` (array of `{value, to}`) — routing table
- `default_to` (string) — fallback target when no case matches

```yaml
edges:
  - from: "classifier"
    to: "billing_handler"
    mode: "switch"
    when: "outputs.classifier.category"
    cases:
      - value: "billing"
        to: "billing_handler"
      - value: "technical"
        to: "tech_handler"
      - value: "sales"
        to: "sales_handler"
    default_to: "human_agent"
```

### Semantics

- The `when` expression is evaluated to produce a value.
- The value is matched against each case's `value` field (strict equality).
- First match wins — exactly one branch executes (mutual exclusion).
- If no match and `default_to` is set, that target executes.
- If no match and no `default_to`, the switch edge is a no-op (workflow continues without branching).
- The `to` field on the edge itself serves as the primary target for schema compatibility.

### Schema Change

```json
"cases": { "type": "array", "items": { "type": "object", "required": ["to"], "properties": { "value": {}, "to": { "type": "string" } } } },
"default_to": { "type": "string" }
```

Added `"switch"` to edge `mode` enum. Added `cases` and `default_to` to `$defs/edge/properties`.

## 5. Sub-Workflows

### Specification

Add two optional fields to node definitions:

- `workflow_ref` (string) — URI or id of an imported sub-workflow
- `workflow_inputs` (object) — Input parameters passed to the sub-workflow

```yaml
imports:
  - "./etl-pipeline.osop"

nodes:
  - id: "run_etl"
    type: "system"
    purpose: "Execute ETL sub-workflow"
    workflow_ref: "etl-pipeline"
    workflow_inputs:
      dataset: "${inputs.dataset}"
      target_table: "analytics.events"
```

### Semantics

- The referenced workflow must be listed in `imports` (or resolvable by the runtime).
- The sub-workflow executes as a single atomic node from the parent's perspective.
- Sub-workflow outputs are accessible as `outputs.<node_id>.*`.
- Sub-workflow failures propagate to the parent node (which may have its own `retry_policy` or `fallback` edges).
- Nesting depth is implementation-defined but SHOULD be capped at 10 levels.
- Circular references (A → B → A) MUST be rejected at validation time.

### Schema Change

```json
"workflow_ref": { "type": "string" },
"workflow_inputs": { "type": "object", "additionalProperties": true }
```

Added to `$defs/node/properties`.

## Backwards Compatibility

All changes are additive:

- New fields are optional with sensible defaults.
- The edge `mode` enum gains one value (`switch`); existing values unchanged.
- Files with `osop_version: "1.0"` remain valid under the v1.1 schema.
- Runtimes that don't support v1.1 features will encounter unknown fields and SHOULD ignore them gracefully (per the `additionalProperties` / `patternProperties` convention).

---

## Migration Guide

No migration needed. To opt in to v1.1 features:

1. Change `osop_version: "1.0"` to `osop_version: "1.1"`.
2. Use the new fields as needed.
3. Existing workflows continue to work unchanged.
